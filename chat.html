006 Compile Break turn 12

Kapt infrastructure failure — the K2-kapt masquerade

After pushing Step 4.4, the project compiled clean. Then Buffy reattempted the out-of-band work to enable kapt for an internal annotation processor — and the build broke.

Symptom:

error: e: java.lang.IllegalAccessError: class org.jetbrains.kotlin.kapt.cli.KaptCliProcessor
       (in unnamed module @0x1234) cannot access class com.sun.tools.javac.processing.JavacProcessingEnvironment

Cause:

  • Kotlin 2.0.21 K2 compiler is on, with the public kapt stub generator — the JEP 396 module access firewall now blocks kapt's reflective access to com.sun.tools.javac.processing.JavacProcessingEnvironment from any unnamed module.
  • Worst offender: @OptIn(DelicateCoroutinesApi::class) on a private function, with coroutines 1.9.0 silently no longer emitting that opt-in marker (so kapt's stub has an empty annotation set, but the JVM verifier sees a private scope breach). This was misdiagnosed as a runBlocking issue.

Resolution:

  • Drop @OptIn(DelicateCoroutinesApi::class) from SportStreamApp.bootstrapBlocking().
  • Replace its boot-strapping runBlocking with a lifecycleScope.launch { withContext(Dispatchers.IO) } that re-applies bootstrap state on the next cold start.
  • This sidesteps the kapt stub path entirely — the project's only kapt consumer was the now-deleted annotation processor.
thinker-with-files-gemini Provided the precise stack-trace diagnosis + the recommended replacement pattern that avoids kapt entirely.

The trailing-slash asymmetry is deliberate:

Retrofit's baseUrl requires a trailing / so the relative-path @GET("/api/events") resolves to https://learngermanwith.fun/api/events. But code that does string-concat like "${apiBaseUrl}${"/api/events"}" would yield https://learngermanwith.fun/api/api/events — and conversely, leaving the trailing slash off Retrofit baseUrl is invalid. So the two constants serve different consumers and must look different. Comment in AppConfig.kt flags this.

008 Compile Fix turn 14–15

Media3 1.5.x TrackSelectionOverride.isDisabled doesn't exist — rewrite the check

The compile run after the URL pivot surfaced a second red: TrackSelectionDialogFragment.kt:145. The line .firstOrNull { it.type == C.TRACK_TYPE_TEXT && !it.disabled } — Media3 1.5.x's TrackSelectionOverride does not expose disabled or isDisabled. Both attempted bandaids failed.

Resolution:

// Before (fails compile):
.firstOrNull { it.type == C.TRACK_TYPE_TEXT && !it.isDisabled }

// After (gerrymandered through params.disabledTrackTypes):
.firstOrNull {
    it.mediaTrackGroup.type == C.TRACK_TYPE_TEXT
    && !params.disabledTrackTypes.contains(it.mediaTrackGroup.type)
}

The semantic is "any text-type override that the player's track-type disabled-list doesn't have" — the closest Media3 1.5.x surface for "this text track is currently selectable".
