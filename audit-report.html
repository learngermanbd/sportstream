🏗️ 2. Architecture

The codebase follows a three-layer MVVM/Clean architecture with Kotlin Coroutines, Room for local persistence, and OkHttp for remote calls.

Layer breakdown

✓ UI layer — mature

5 Activities (Splash, Main, Player, Update, Crash) · 8 ViewModels (Home, Categories, Favorites, Highlights, Player, Main, Search, Notice, NetworkStream) · dedicated adapters and base StateViewModel with CancellationException-safe scope.

✓ Data layer — production-ready

3 Repositories (Main, Favorites, Playlist) with thin-pass-through to RemoteDataSource / LocalDataSource. DI seams via NetworkModule + LocalModule + RepositoryModule.

✓ Security layer — exceptional depth

30 dedicated security files covering 20+ protection mechanisms. Each module is documented inline with Phase/Step provenance (e.g. Phase 7 · Step 7.4). Excellent traceability.

The native layer is properly compiled (libsportstream_security.so present in APK for arm64-v8a / armeabi-v7a / x86_64). The opaque-export trick via JNI_OnLoad + RegisterNatives makes static analysis harder for attackers.

⚠️ 6. Gaps & Risks

The following issues should be addressed before a public release.

🚨 [CRITICAL] Zero automated testing

Neither app/src/test (JUnit) nor app/src/androidTest (Espresso/UI) directories exist. No testImplementation dependencies in build.gradle.kts. The security scoring logic, repository code paths, and ViewModel state transitions are completely unverified in CI.

🚨 [CRITICAL] Missing signing.properties

The signing.properties file is not in the repo (correctly — credentials shouldn't be committed). However, the build script falls back to debug signing for release if the file is absent. This means a CI run without proper signing config produces a release APK signed with the debug keystore — which is acceptable for demos but dangerous if accidentally deployed.

🚨 [CRITICAL] EncryptedConstants XOR weakness

The build-time encryptSecrets task generates EncryptedConstants.kt with AES-256-GCM ciphertext, but downstream code uses XOR to combine key halves in plain Kotlin. A Frida/IDA hook on the XOR step trivially recovers the base URL — undermining the runtime protection. Limits protecting only against static string dumps, not dynamic instrumentation.

⚠️ [HIGH] Magisk Hide / Zygisk evasion risk

The native root check searches /data/adb/magisk via stat(). RootHide, Zygisk denylist, or Shamiko-equipped Magisk will hide the directory. Consider cross-referencing /proc/self/mountinfo for overlay mounts and checking ro.debuggable + SELinux contexts.

⚠️ [HIGH] No release-mode UI verification

The codebase has never been smoke-tested end-to-end on a real device in release mode (R8 obfuscation may strip a class name that Glide or reflection depends on). Recommendation: adb install app-release.apk on a clean Android 12+ device and verify all 5 activities boot without ClassNotFoundException.

⚠️ [MEDIUM] SYSTEM_ALERT_WINDOW trust check

The SYSTEM_ALERT_WINDOW permission is requested but the app does not verify it was actually granted at runtime. On Android 12+ the user must explicitly grant via Settings — a missing grant would silently crash the floating player service.

ℹ️ [MEDIUM] Pin-set is static

SSL pins are baked into network_security_config.xml with an expiration of 2027-01-01. If Let's Encrypt rotates the intermediate CA ahead of schedule, all users will get connection failures. Consider adding a backup pin rotation mechanism (fetched from /api/config every 30 days via RemoteConfigHelper).

ℹ️ [MEDIUM] Native crash propagation

If native_security.so panics (e.g. unmapped memory on a tamper attempt), the isolated :crash process and main process both die simultaneously because the JVM-level signal handler doesn't differentiate. Consider registering a sigaction that escalates gracefully.

ℹ️ [LOW] Inconsistent ProGuard keep rules

Some @SerializedName-driven Gson models may have their field names obfuscated. The 107-line proguard-rules.pro covers Media3 and Glide but doesn't explicitly keep all data model constructors. Fastjson-style serialization breakage is possible unless verified.

✅ 8. Recommendations

Before next commit

  1. Add a JUnit test for SecurityGate scoring — at minimum unit tests for the score thresholds. Without tests, refactoring is dangerous.
  2. Refactor signing.properties fallback to fail the build if signing.properties is missing AND CI flag is set — never silently sign release with debug key.
  3. Verify release APK installs on a real device. Use adb install -r app-release.apk and walk all 5 activities.
  4. Add testImplementation deps (JUnit 5 + Truth + Mockito-Kotlin). Free-tier library, no runtime cost.

Before production release

  1. Strengthen anti-root logic with /proc/self/mountinfo overlay detection and a tied ro.debuggable + SELinux check.
  2. Add runtime pin rotation by reading backup pin set from /api/config every 30 days.
  3. Migrate encrypted secrets to pure JNI — XOR step should happen inside libsportstream_security.so, not in cleartext Kotlin.
  4. Add SAFEGUARD in CrashActivity for native signum propagation — surface a recovery screen instead of letting process die.
  5. End-to-end Playwright/CI test for the whole login → stream → favorite → notification pipeline once Phase 8 backend lands.
